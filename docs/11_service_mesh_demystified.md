# 第十一章：Service Mesh 祛魅

## 1. 东西流量 vs 南北流量

### 1.1 南北流量（North-South）
**定义**：客户端到服务器的流量。
```
Client → Gateway → Service
```
**特点**：需要经过防火墙、负载均衡器。

### 1.2 东西流量（East-West）
**定义**：服务到服务的流量。
```
User Service → Order Service → Payment Service
```
**特点**：在数据中心内部流动，频率极高。

---

## 2. Sidecar 模式的推导

### 2.1 痛点：每个 Gin 都要写重试、熔断代码
**场景**：服务 A 调用服务 B，需要处理：
- 超时重试
- 熔断器
- 链路追踪

**问题**：每个服务都要写一遍这些代码，重复劳动。

### 2.2 解法：把网络逻辑剥离到 Sidecar
**核心思想**：每个服务旁边运行一个代理（Envoy）。
```
+------------------+       +------------------+
|   User Service   |       |   Order Service  |
+------------------+       +------------------+
        ↓                           ↓
+------------------+       +------------------+
|   Envoy Sidecar | ←──→  |   Envoy Sidecar |
+------------------+       +------------------+
```

**优势**：
- 服务只需关注业务逻辑
- 网络问题由 Sidecar 统一处理
- 支持多语言服务（Sidecar 是独立的进程）

## 4. 对 Gin 程序员的意义

### 4.1 你的代码不需要改
**以前**：在 Gin 中写重试逻辑。
```go
// ❌ 不推荐
resp, err := http.Get("http://order-service/orders")
if err != nil {
    // 重试 3 次
}
```

**现在**：让 Sidecar 处理。
```go
// ✅ 推荐
resp, err := http.Get("http://order-service/orders")
// Sidecar 会自动重试
```

### 4.2 只需关注业务
**你的职责**：
- 实现 API 逻辑
- 处理业务逻辑
- 返回正确数据

**Sidecar 的职责**：
- 服务发现
- 负载均衡
- 超时重试
- 链路追踪

## 5. 主流 Service Mesh 方案

### 5.1 Istio
- 功能最全
- 复杂度高
- 适合大规模集群

### 5.2 Linkerd
- 轻量级
- 性能好
- 适合中小规模集群

### 5.3 Consul Connect
- 集成服务发现
- 简单易用
